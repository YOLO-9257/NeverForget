## 🔵 Phase 1: 强基阶段

> **目标**: 打好产品基础，提升应用形态，完善安全体系  
> **预计周期**: 4-6 周

### 1.1 PWA (Progressive Web App) 支持

**优先级**: 🔴 高

**描述**: 将 Web 应用打包为可安装的应用程序，提供接近原生 App 的用户体验。

**功能清单**:

- [ ] 配置 `manifest.json` 文件
  - 应用图标 (多尺寸)
  - 应用名称和描述
  - 主题色和背景色
  - 启动画面配置
- [ ] 实现 Service Worker
  - 静态资源缓存策略
  - 离线页面支持
  - 后台同步能力
- [ ] 添加安装引导
  - 首次访问提示安装
  - 设置页面的安装入口
- [ ] 实现 WebPush 推送
  - 浏览器推送订阅
  - 推送权限管理
  - 离线消息队列

**技术方案**:
```
admin/
├── public/
│   ├── manifest.json          # PWA 清单
│   ├── sw.js                  # Service Worker
│   └── icons/                 # 应用图标
└── src/
    └── utils/
        └── pwa.ts             # PWA 工具函数
```

## 🟢 Phase 2: 扩展阶段

> **目标**: 扩展通知能力，融入用户现有生态  
> **预计周期**: 6-8 周

### 2.1 多通道通知系统

**优先级**: 🔴 高

**描述**: 构建统一的 Notification Provider 抽象层，支持多种推送通道。

**功能清单**:

- [ ] 通知渠道抽象层
  ```typescript
  interface NotificationChannel {
    id: string;
    name: string;
    send(message: NotificationMessage): Promise<SendResult>;
    validateConfig(config: ChannelConfig): boolean;
  }
  ```

- [ ] 邮件通道 (Email)
  - 支持 SendGrid / Mailgun / Resend
  - HTML 邮件模板
  - 附件支持

- [ ] Telegram Bot
  - Bot Token 配置
  - 富文本消息格式
  - 内联按钮交互

- [ ] Discord Webhook
  - Webhook URL 配置
  - Embed 消息格式
  - 角色提醒

- [ ] 飞书 (Lark)
  - 应用机器人
  - 卡片消息

- [ ] 浏览器推送 (WebPush)
  - VAPID 密钥配置
  - 订阅管理
  - 推送通知

**数据模型扩展**:
```sql
-- 用户通知渠道配置表
CREATE TABLE notification_channels (
    id TEXT PRIMARY KEY,
    user_key TEXT NOT NULL,
    channel_type TEXT NOT NULL,  -- 'email', 'telegram', 'discord', 'lark', 'webpush'
    channel_name TEXT NOT NULL,  -- 用户自定义名称
    config TEXT NOT NULL,        -- JSON 配置
    is_default INTEGER DEFAULT 0,
    status TEXT DEFAULT 'active',
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

-- 提醒关联的通知渠道 (多对多)
CREATE TABLE reminder_channels (
    reminder_id TEXT NOT NULL,
    channel_id TEXT NOT NULL,
    PRIMARY KEY (reminder_id, channel_id)
);
```

### 2.2 日历订阅 (iCal/ICS)

**优先级**: 🟡 中

**描述**: 生成标准的 iCal 订阅链接，让用户可以在系统日历中查看 NeverForget 的任务。

**功能清单**:

- [ ] ICS 文件生成
  - VEVENT 事件格式
  - RRULE 重复规则
  - VALARM 提醒规则
- [ ] 订阅链接
  - 用户专属订阅 URL
  - Token 鉴权
  - 增量更新支持
- [ ] 双向同步 (高级)
  - CalDAV 协议支持
  - Google Calendar 集成
  - Apple Calendar 集成

**API 设计**:
```http
GET /api/calendar/subscribe?token=xxx
Content-Type: text/calendar

# 返回 ICS 格式的日历数据
```

## 🟣 Phase 3: 体验阶段

> **目标**: 降低使用门槛，提升交互体验  
> **预计周期**: 8-10 周

### 3.1 自然语言输入

**优先级**: 🔴 高

**描述**: 支持用户使用自然语言描述提醒，自动解析为调度规则。

**功能清单**:

- [ ] 中文自然语言解析
  - "明天下午3点提醒我开会"
  - "每周五下午5点下班提醒"
  - "每月15号还信用卡"
  - "下周一上午9点面试"
- [ ] 英文自然语言解析
  - "Remind me to buy milk tomorrow at 5pm"
  - "Every Monday at 9am"
- [ ] 智能纠错和确认
  - 模糊时间确认 (如 "下午" -> 具体时间)
  - 解析结果预览
  - 手动调整入口

**技术方案**:
- 方案 A: 本地规则引擎 (chrono-node/dayjs 扩展)
- 方案 B: LLM API 解析 (OpenAI/Claude)
- 方案 C: 混合方案 (本地 + LLM 兜底)

## 🔧 技术债务

> 在功能迭代的同时，需要持续解决的技术问题

### 高优先级

- [ ] **时区处理统一化**: 抽取时区逻辑到独立模块，避免散落在各处
- [ ] **错误处理增强**: 统一错误码，完善错误日志
- [ ] **API 版本管理**: 引入 API 版本控制 (`/api/v1/`, `/api/v2/`)

### 中优先级

- [ ] **测试覆盖率**: 添加单元测试和集成测试
- [ ] **性能监控**: 接入 APM 工具 (Sentry/Datadog)
- [ ] **文档完善**: API 文档自动生成 (OpenAPI/Swagger)

### 低优先级

- [ ] **国际化 (i18n)**: 支持多语言界面
- [ ] **无障碍 (a11y)**: 符合 WCAG 标准
- [ ] **代码重构**: 模块化拆分，优化代码结构
